## Color

Phase 3 introduces a dual-theme color system. All colors are delivered via CSS custom properties on `:root` (or a `[data-theme]` attribute on `<html>`). Canvas renderers read theme colors at render time from a theme state object.

### Theme Architecture

Theme is stored as a `$state` value (`'dark' | 'light'`). On change:
1. `data-theme` attribute is set on `<html>` element
2. CSS custom properties update via `[data-theme="dark"]` / `[data-theme="light"]` selectors
3. Canvas render functions receive a theme colors object (not CSS vars -- canvas cannot read CSS custom properties)

## Interaction Contract

### Theme Toggle (UI-04, UI-05, D-05, D-06, D-07, D-08)

| Property | Value |
|----------|-------|
| Position | `position: absolute; top: 8px; right: 8px` (D-05: top-right, small, out of way) |
| Element | `<button>` with `aria-label="Toggle theme"` |
| Icon | Inline SVG: sun icon (16px) in light mode, moon icon (16px) in dark mode (D-06) |
| Hit area | 44px x 44px minimum (touch-friendly) |
| Visual size | 32px x 32px button with centered 16px icon |
| Background | transparent; no visible button chrome |
| Icon color | `var(--color-text-muted)` at rest |
| Icon hover | `var(--color-text)` on hover/focus |
| Transition | Theme switch is instant -- no CSS transition on theme change (simplicity first) |
| Icon crossfade | 150ms opacity crossfade between sun and moon icons |
| Z-index | 10 (same layer as other overlays) |
| Default | Dark theme on first visit (D-08) |
| Persistence | `localStorage.setItem('theme', 'dark' | 'light')` (D-07, UI-05) |
| On load | Read `localStorage.getItem('theme')`; if null, default to `'dark'` |
| pointer-events | auto (clickable at all times, including during game over) |

### Touch: Tap to Place (UI-06, D-09)

| Property | Value |
|----------|-------|
| Trigger | `touchend` after a quick touch: duration < 200ms AND total movement < 10px (D-09) |
| Movement threshold | 10px cumulative (sum of abs deltas from touchstart position) |
| Time threshold | 200ms between `touchstart` and `touchend` |
| Behavior | Convert `touch.clientX/clientY` to canvas offset, then same hex resolution as mouse click |
| Conflict with pan | If movement exceeds 10px OR duration exceeds 200ms, treat as pan (not tap) |
| Hover preview | No hover preview on touch (no `mousemove` equivalent; touch has no "hover" state) |
| Game status check | Only place if `gameState.status === 'playing'` |
| Prevent default | `touchstart` calls `preventDefault()` to suppress 300ms click delay and prevent scroll |

### Touch: Drag to Pan (UI-07, D-11)

| Property | Value |
|----------|-------|
| Trigger | Single-finger `touchmove` that exceeds 10px movement threshold |
| Behavior | Camera translates by touch delta in world space (same math as mouse pan) |
| Touch tracking | Track `touch.identifier` from first touch in `touchstart` to handle multi-touch correctly |
| Inertia | none (consistent with mouse pan from Phase 1) |
| Game over | Pan disabled when `gameState.status === 'won'` (inherited from Phase 2) |

### Touch: Pinch to Zoom (UI-07, D-10)

| Property | Value |
|----------|-------|
| Trigger | Two-finger touch (`touches.length === 2`) |
| Zoom anchor | Midpoint between the two touch points (D-10: zoom toward center of fingers) |
| Zoom calculation | New distance / initial distance between fingers = scale factor |
| Zoom min/max | Same as scroll zoom: min 0.4, max 3.0 |
| Smoothness | Apply scale delta each `touchmove` frame (continuous, not stepped) |
| Pan during pinch | Midpoint translation applies pan simultaneously (standard map behavior) |
| Prevent default | `touchmove` with 2+ touches calls `preventDefault()` to prevent page zoom |
| Game over | Zoom disabled when `gameState.status === 'won'` |

### Mouse Interactions (inherited from Phase 1/2, unchanged)

All existing mouse interactions (click-to-place, hover preview, drag-to-pan, scroll-to-zoom, debug toggle) remain unchanged. Touch handlers are additive, not replacements.

## Theme Color Object (Canvas Renderer Interface)

Canvas 2D API cannot read CSS custom properties. The renderer receives a plain object:

```typescript
interface ThemeColors {
  background: string;        // '#1a1a2e' | '#f5f0e8'
  gridStroke: string;        // 'rgba(255,255,255,0.12)' | 'rgba(0,0,0,0.10)'
  playerX: string;           // '#4fc3f7' | '#0288d1'
  playerO: string;           // '#ef5350' | '#c62828'
  playerXHover: string;      // 'rgba(79,195,247,0.3)' | 'rgba(2,136,209,0.25)'
  playerOHover: string;      // 'rgba(239,83,80,0.3)' | 'rgba(198,40,40,0.25)'
  debugText: string;         // 'rgba(255,255,255,0.5)' | 'rgba(0,0,0,0.4)'
  lodDot: string;            // 'rgba(255,255,255,0.2)' | 'rgba(0,0,0,0.15)'
  rejectionFlash: string;    // 'rgba(255,80,80,0.4)' | 'rgba(198,40,40,0.35)'
  edgeFadeOpaque: string;    // 'rgba(26,26,46,1.0)' | 'rgba(245,240,232,1.0)'
  edgeFadeTransparent: string; // 'rgba(26,26,46,0.0)' | 'rgba(245,240,232,0.0)'
}
```

Two static objects (`DARK_THEME` and `LIGHT_THEME`) are defined once and selected by theme state. The renderer function signature gains a `colors: ThemeColors` parameter replacing all hardcoded color constants.

## Canvas Rendering Contract (updated from Phase 2)

### Render Order (unchanged)

Same 10-step pipeline as Phase 2. The only change is that all color values come from the `ThemeColors` object instead of module-level constants.

### Performance Budget (extended)

| Metric | Target |
|--------|--------|
| Theme switch | < 1 frame (16ms) -- swap color object reference + trigger redraw |
| Touch event processing | < 2ms per touch event (coordinate conversion + state update) |
| Pinch zoom per frame | < 4ms (two-point distance calc + camera update + redraw trigger) |
